## 討論內容

1. AI 分工流程：
   - 先用 ChatGPT 做高層產品規劃（功能/流程/架構），再整理成結構化 prompt 交給 Codex 產生實作。
   - 優點：降低 Token 浪費（規劃用 ChatGPT、實作用 Codex）、提高可控性與生成品質。

2. AI 最多做到 80 分，但剩下的 20 分必須靠人來完成
   - 剩下 20 分需要人來補齊與打磨（優化內容、新增機制、修正不符需求、細節客製、反覆迭代）。
   - 在專題中，需要能清楚區分：AI 提供的內容 vs 團隊自己的判斷、修改與取捨。

3. 教授用馬斯克的例子說明：先提出 100 分的想像，即使最後只做到 20 分，也可能足以改變世界，只要能突破框架、做出核心價值仍有份量。

4. 6/2 文件撰寫方向：先寫目前最具體可行的工具/系統設計/技術規劃與開發流程；最終成果允許與文件規劃不同，隨工具迭代而進化，做得更好反而加分。

5. GitHub 經營要求：從現在起持續累積與推送（每次會議紀錄、Keras 實作、簡報、文件版本、重要修改與進展），作為專題歷程證據。

## 決議事項

- 採用「ChatGPT 規劃 → Codex 實作」的 AI 分工方式，並將規劃內容整理成結構化 prompt，後續專題評審將需要用到。
- 後續所有 AI 產出需由團隊持續修正/客製化，並在成果/報告中能說明 AI 與人為貢獻的分界。
- 專題以高標願景為目標，但以「做出核心價值、可用成果」作為落地評估重點。
- 6/2 交付文件先以當下最具體可行方案撰寫，保留後續因工具迭代而調整的彈性。
- GitHub 立即開始持續更新：會議紀錄、實作/實驗、簡報、文件與版本推進紀錄都要上傳留痕。
